# Pico Scheduler Firmware Generator Design

## Goal

Replace the generic Pico scheduler runtime-plus-`state.json` deployment model with a generated, controller-specific `main.py` firmware file that is easy to inspect and easy to extend.

The generator should:

- keep the host-side JSON state as the source of truth
- generate a single concrete `main.py` per Pico scheduler controller
- copy only that `main.py` to the Pico
- avoid unused code for unsupported device types
- establish a clean pattern for future firmware families such as `pico_dosimeter`

## Problem

The current Pico scheduler deployment model is more generic than the project needs:

- a shared handwritten `pico_scheduler/main.py` is copied to every Pico
- a second file, `state.json`, is also copied to the Pico
- the Pico interprets JSON at boot
- the runtime always contains both GPIO and PWM code paths even if one is unused
- the runtime structure is harder to understand because the interesting schedule state is data-driven at runtime rather than visible as concrete code

This makes the device-side implementation harder to inspect and pushes complexity onto the Pico that can instead live on the host.

## Design

### Source of truth

The host remains the source of truth.

The existing validated scheduler state on the Raspberry Pi stays in the current JSON shape:

```json
{
  "report_every": 5,
  "devices": [
    {
      "type": "gpio",
      "pin": 15,
      "current_t": 0,
      "reschedule": 1,
      "pattern": [
        {"val": 1, "dur": 10},
        {"val": 0, "dur": 20}
      ]
    }
  ]
}
```

That JSON is no longer copied to the Pico. Instead, it is used as generator input on the host.

### Generated firmware model

For each `pico_scheduler` controller, the host generates one concrete `main.py`.

The generated firmware should:

- declare device-specific variables in code
- bake in `report_every`
- bake in the schedule patterns
- run a simple loop
- emit full state reports periodically
- avoid runtime JSON parsing entirely

The generated code should be understandable without reading generator internals. Template output should look like ordinary MicroPython rather than a compressed code blob.

### Firmware provenance block

Generated `main.py` should start with a top-level triple-quoted string containing generator provenance and the exact generator input.

Example shape:

```python
"""
Generated by: pico_scheduler_generator
Generated at: 2026-05-04T13:02:11-10:00
Git version: 3486792
Controller: pump_n_lights
Generator input:
{
  "report_every": 5,
  "devices": [
    {
      "type": "gpio",
      "pin": 15,
      "current_t": 0,
      "reschedule": 1,
      "pattern": [
        {"val": 1, "dur": 10},
        {"val": 0, "dur": 20}
      ]
    }
  ],
  "options": {
    "loop_sleep_ms": 20,
    "pwm_freq": 1000
  }
}
"""
```

Rules:

- include generator name
- include generation timestamp
- include `git_version`
- include controller id
- include generator input as formatted JSON
- do not put this metadata into runtime reports

### Generator structure

The generator should be host-side Python code with templates.

Recommended structure:

- `pico_scheduler/generator.py`
  - public entrypoint for generating firmware text
  - takes controller id, validated state, and generation options
- `pico_scheduler/templates/`
  - base template
  - optional GPIO template fragment
  - optional PWM template fragment
  - optional shared helper fragments

The templates should include short explanatory comments and concrete examples where they materially help readability.

The generator should prefer clear template composition over opaque string concatenation.

### Generation options

The generator should accept a small explicit options object with good defaults.

Initial options:

- `loop_sleep_ms`, default `20`
- `pwm_freq`, default `1000`

Rules:

- options belong to the generator, not the Pico runtime protocol
- unused options must not cause dead code to be emitted
- options should be recorded in the firmware provenance block

### Selective code emission

Generated firmware should contain only the code needed by the controller state.

Rules:

- if there are no GPIO devices:
  - no `Pin` usage for GPIO outputs
  - no GPIO-only helpers
- if there are no PWM devices:
  - no `PWM` import
  - no PWM setup code
  - no PWM-only helpers
- shared timing and report logic may remain shared if still clear

The goal is not extreme minimization at any cost. The goal is obvious, readable output with no irrelevant device-family code.

### Runtime message contract

Generated Pico firmware should use `type`, not `kind`.

Supported message types:

- `report`
- `error`

There is no `startup` message in generated firmware.

Every `report` is the full current scheduler state snapshot.

Example:

```json
{
  "type": "report",
  "content": {
    "devices": [
      {
        "type": "gpio",
        "pin": 15,
        "elapsed_t": 12,
        "cycle_t": 12,
        "reschedule": 1,
        "pattern": [
          {"val": 1, "dur": 10},
          {"val": 0, "dur": 20}
        ],
        "current_value": 0
      }
    ]
  }
}
```

Rules:

- keep the wire format minimal
- do not include provenance metadata in reports
- preserve the existing state-oriented `devices` payload concept
- `error` messages remain explicit structured failures for the host monitor

### Deployment flow

The current apply path already has the right seam: it copies firmware artifacts to the Pico and resets it.

That flow should change to:

1. validate host-side scheduler state
2. generate temporary `main.py` text for the controller
3. copy generated `main.py` to the Pico as `:main.py`
4. reset the Pico

For generated `pico_scheduler` firmware:

- do not copy `state.json`
- do not depend on `state.json` at boot
- keep the host-side timer JSON in `data/timers/` as the durable source of truth

Re-applying controller state means re-generating firmware and copying the new `main.py`.

### Server impact

The server should stop treating `pico_scheduler/main.py` as the exact firmware payload copied to the device.

Instead, the server should:

- use the generator to render controller-specific firmware
- continue validating and saving host-side timer state
- continue parsing live `report` messages into host-side monitor state

The server may keep compatibility when reading older live reports during transition, but the generated-firmware target contract should be:

- `type`, not `kind`
- `devices`, not `events`
- no `startup`

### Future firmware families

This slice establishes a per-firmware-family generator pattern.

That pattern should support:

- `pico_scheduler` generator and templates now
- later `pico_dosimeter` generator and templates

Future families should not be forced through scheduler-specific code paths. Shared utilities are acceptable, but the generator API should keep firmware family boundaries clear.

## Compatibility

This is a deliberate deployment-model change, so compatibility should be handled at the host monitor boundary, not by keeping the Pico deployment path half-generic forever.

Acceptable transition behavior during implementation:

- host monitor accepts both `kind` and `type` if needed
- host monitor accepts both old and new generated reports while the branch is in progress

Final target for generated firmware:

- no `state.json`
- `type` message envelope
- only `report` and `error`

## Impacted areas

- `pico_scheduler/README.md`
- `pico_scheduler/main.py` or its replacement role in the deployment flow
- new generator module(s) under `pico_scheduler/`
- new template files under `pico_scheduler/templates/`
- `plamp_web/server.py`
- tests covering generator output and Pico apply behavior

## Testing requirements

Required coverage:

- generator renders readable `main.py` for a GPIO-only controller
- generator renders readable `main.py` for a PWM-only controller
- generator omits PWM imports/code for GPIO-only input
- generator omits GPIO-only code for PWM-only input
- generated file contains the provenance triple-quoted block with generator name, timestamp, `git_version`, controller id, and input JSON
- apply path copies generated `main.py` and does not copy `state.json`
- server monitor accepts generated `type: report` messages
- existing host APIs and UI continue to work with generated firmware reports

## Non-goals

- redesign host-side scheduler JSON shape
- add semantic firmware versioning separate from git provenance
- add firmware-side support for new controller families in this slice
- optimize for smallest possible generated source at the expense of readability
